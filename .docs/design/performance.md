# Performance

## Scope
- In: numeric budgets, caches that defend them, telemetry hooks.
- Out: rendering pipeline details (see `features/rendering.md`), persistence batching specifics (see `features/persistence.md`).

## Budgets

Spec §15 gates, not targets. CI asserts each via `cargo xtask bench`. Phase 17.9 added a **variance-tail contract** alongside p99: every gate also asserts `p99.9 ≤ 2 × p99_budget` via `continuity_test_support::percentiles::assert_within_budget`. Sample distributions print on every run with the p99/p50 jitter ratio so sporadic outliers are visible even when the headline gate passes.

| Path | p99 | Enforced by |
|---|---|---|
| Keystroke → pixel | ≤ 8 ms | bench harness, `perf_gate_*` tests (sub-table below) |
| Edit → durable | ≤ 400 ms | `continuity-persist::perf_gates::perf_gate_edit_to_durable_under_400ms` |
| Cold start (empty session) | ≤ 120 ms | startup benchmark |
| Find in 200 buffers / 50 MB total | ≤ 80 ms | `continuity-search::perf_gates` (per-buffer scoped post-§G6) |
| Stripped binary | ≤ 8 MB | `xtask bench` binary-size check |
| Steady-state per-keystroke heap alloc | 0 bytes (aspirational) | not yet gated — see hot-path discipline below |
| Memory ceiling (empty / 50 buffers / 200 buffers) | ≤ 40 / 90 / 180 MB | `continuity-test-support::perf_gates_memory_{empty,50,200}` |

### Keystroke → pixel sub-table (Phase 17.9 §B)

The 8 ms budget breaks down by stage; each stage is gated independently so a paint regression names its own slice.

| Slice | p99 | Gate |
|---|---|---|
| core apply_edit | ≤ 4 ms | `continuity-core::perf_gates::perf_gate_core_apply_p99_under_4ms` |
| decoration incremental parse | ≤ 1 ms | `continuity-decorate::perf_gates::perf_gate_decoration_incremental_parse_p99_under_1ms` |
| D2D draw submission (Present excluded) | ≤ 2 ms | `continuity-render::perf_gates::perf_gate_d2d_draw_submission_p99_under_2ms` |
| WM_PAINT → frame-ready (submission + Present) | ≤ 2 ms | `continuity-ui::perf_gates::perf_gate_wm_paint_to_frame_ready_p99_under_2ms` |

The Present-only slice is folded into the WM_PAINT gate rather than gated separately — Present(0) on a flip-discard chain has GPU-bound tail variance independent of CPU submission cost; isolating Present requires a different methodology (frame-pacing instead of pumping). Gate as one for now.

### Perf history

`cargo xtask perf-snapshot` collects p50/p95/p99/p99.9/jitter per gate and writes `target/perf/snapshot-<sha>.json`. `cargo xtask perf-history-append` appends to the tracked `.perf/history.jsonl` (idempotent on `(sha, host_id)`). `cargo xtask perf-report` prints a per-gate trend table; `perf-compare --baseline <sha>` exits non-zero on >10 % p99 or >20 % p99.9 growth. CI runs `perf-history-append` on every push to `main` and `perf-compare` on every PR (sticky comment, non-blocking initially). Long-form reference: `.docs/technical/testing-and-perf.md`.

### E2E perf samples

The `Win32Harness` (`crates/test_support/src/win32_harness.rs`) drives the production wndproc + Renderer through a real hidden HWND. The §C E2E tests in `crates/ui/tests/` (smoke / pane split / multi-window / settings live-reload) exercise the assembled paint pipeline end-to-end. The WM_PAINT gate uses the same harness pattern internally so a paint regression that would surface in the e2e suite also trips the gate at commit time.

### Current paint baseline

P18's large-buffer arc replaced paint-time worker waiting with structural fast paths: viewport-priority partial row-index walks, width-independent segment shaping, larger segment-cache capacity, partial-fill dedupe, and a single non-blocking worker poll per paint. Verification trace `perf-snapshots/trace_20260520-231028.report.md` over a 124,796-row session reported zero `stall100` rows and these maxima:

| Label | Max |
|---|---|
| `frame_display:cache_hit` | 154 µs |
| `frame_display:dirty_rebuild` | 168 µs |
| `frame_display:viewport_realize` | 57 µs |
| `frame_display:worker_hit` | 5.1 ms |

`event:worker_wait`, `event:worker_resubmit`, and `event:paint_loading_overlay` are extinct in current traces. A trace containing any of those events predates the current no-wait paint path or has stale instrumentation.

Same-geometry `realized_miss_partial` frames also seed the cache because they only extend a compatible frame's realized row window. |

## Caches that defend stale results

- **Revision stamp on every async result.** Drop on revision mismatch — no callback dance.
- **Coalesce-on-full at persist queue.** Adjacent `Insert`/`Delete` ops within the same `(buffer, undo_group)` merge before a re-try when the queue is full.
- **Decoration newest-wins.** Pool inbox keeps one pending request per buffer; older requests are replaced.
- **Prewarm revision keys.** MRU display-map prewarm entries are dropped on rope-revision drift and on newer decoration revisions; undecorated viewport entries are allowed only as a temporary paint fallback while the decoration worker catches up.

## Hot-path discipline
- **Zero heap allocations per keystroke (steady state).** Aspirational target; not yet gated by an automated test. `format!`, `to_string`, ad-hoc `Vec::new()` on the keystroke path are violations. A dhat-based gate is tracked as future work — see `unwired_features.md` if a follow-up entry has been filed.
- **No `Mutex` on the keystroke path.** Locks live only at theme / font cache.
- **No regex compile on the keystroke path.** Patterns precompiled once and reused (`grep-regex::RegexMatcher` cached on the find bar).
- **`Arc` clones, not deep clones.** `RopeSnapshot`, `Decorations`, `DisplayMap` all cross thread boundaries by `Arc::clone`.

## Telemetry hooks
- `tracing` integration is a feature flag (`tracing` crate present but optional). Production builds compile it out.
- `dhat` heap profiling: used today for the memory-budget gates in `crates/test_support/tests/perf_gates_memory_*.rs` (empty / 50-buffer / 200-buffer sessions). A zero-alloc-per-keystroke gate is *not* wired yet; the §16 hot-path discipline is enforced by code review.
- `criterion` benches per crate in `benches/`. Run via `cargo xtask bench`.
- `display_prewarm_cache::cached_switch_projection_is_tighter_than_cold_build` verifies the Ctrl+Tab hot path by asserting a prewarmed `FrameDisplay` cache lookup is tighter than building the same projection cold. The idle scheduler also fixes the work budget at one stage per 50 ms tick after a 120 ms input grace.
- Set the `CONTINUITY_PAINT_TRACE` environment variable (any non-empty value) before launching the editor to enable per-stage paint timings from `crates/ui/src/paint_trace.rs`. Each `WM_PAINT` prints one `stage:<name>=<duration>` line per major phase plus a trailing `total=<duration>` line. Built for large-buffer triage — instead of bisecting against synthetic perf gates, drive the editor with the suspect buffer and read the dominant cost off stderr. No overhead when the variable is unset.
- For real-app focus-return triage prefer `CONTINUITY_UI_TRACE=<path>` instead. The trace is TSV: `<ms_since_start>\t<kind>\t<label>\t<duration_us>\t<details>`. `kind` is one of `event`, `paint`, `wndproc`. Every event over 16 ms emits an additional `stall` line; over 100 ms an additional `stall100` line. Grep the log for `\tstall100\t` and `\tstall\t` to find UI-thread blockers without scrolling every line. File-based output survives GUI process stderr redirection that the legacy stderr trace did not.
- **Manual lag trace expansion (2026-05-17).** `CONTINUITY_UI_TRACE=<path>` now emits aggregate diagnostic lines for real interaction passes, not just synthetic perf gates. `paint:projection_stats` carries `source_lines`, full-document `display_rows_total`, viewport row range, realized row range/spec count, `frame_source={worker_hit|cache_hit|inline_selection_reveal|inline_dirty|inline_splice|inline_cold}`, `build_kind`, `worker_miss`, `selection_dirty`, focused image-reservation count, and spectator-pane realized counts. `paint:render_stats` carries layout-cache deltas (`layout_hits`, `layout_misses`, `layouts_created`), focused rows actually drawn (`display_rows_drawn`, `source_lines_visited`, `soft_wrap_continuations`), spectator source lines, and feature toggles/counts (`spell_spans`, `table_layouts`, `image_placements`, `minimap`, `outline`, `status_bar`). Selection-only paths emit `event:selection_set` timing plus `event:selection_update`; mouse drag emits `event:mouse_drag_selection selection_changed=…`; resize emits `event:resize_projection_inputs old/new viewport + wrap width`; every `Window::invalidate` emits `event:invalidate_request`. `event:edit_apply` for `InsertText` adds `input=char|newline|paste|paste_multiline`, so manual traces distinguish typing, raw newline insertion, and paste while command-driven smart newline remains `kind=insert_newline_smart`.
- `crates/ui/tests/perf_gates_large_buffer_paint.rs` exercises the assembled WM_PAINT pipeline at ~6000 source lines via [`Win32Harness`] (`crates/test_support/src/win32_harness.rs`). The caret-move and repeat-paint gates assert p99 ≤ 8 ms and ≤ 4 ms respectively — both held by the last-painted `FrameDisplay` reuse path; a regression that pushes paint back into the cold-build branch trips them within one run.
- `crates/ui/tests/perf_gates_focus_return.rs` drives `WM_ACTIVATEAPP(false)`/`(true)` through the harness against the same 6000-line buffer and measures the first paint plus the first caret-move paint after activation. The 50 ms budget catches a regression that re-introduces synchronous whole-buffer work on the focus path (spell check, decoration re-parse, outline rebuild, …).
- `Window::ensure_spell_fresh` defers the Windows `ISpellChecker` COM sync above 64 KiB; `Window::tick_spell_recheck` (driven from `on_display_prewarm_tick`) hard-refuses to run for buffers above the same cap and leaves `pending_recheck` set so a future worker-thread variant can drain it. The previous behaviour stalled the UI thread for seconds on focus return of a multi-thousand-line natural-text buffer because the COM path scales with the number of misspellings and pays a per-error suggestion fetch. A full worker-thread move is the proper follow-up; until then large-buffer squiggles are stale rather than seconds-blocking.
- **Activation grace.** `Window::in_activation_grace()` returns `true` for `Window::ACTIVATION_GRACE_MS` (1000 ms) after each `WM_ACTIVATEAPP(true)`. Nonessential idle work consults this gate: `prepare_paint_frame` skips `submit_decorations_for_visible_panes` and `ensure_spell_fresh`; `on_display_prewarm_tick` skips MRU prewarm builds entirely. Input + paint dominate during the window; deferred work resumes once the grace expires and the existing prewarm idle gate (120 ms input silence) clears.
- **MRU prewarm size cap.** Buffers at or above `PREWARM_BIG_BUFFER_LINE_CAP` source lines (2000) are skipped from MRU prewarm — each stage cold-builds a full `FrameDisplay` on the UI thread and three stages × two targets adds up to hundreds of milliseconds of idle-tick blocking. The paint-time `last_painted_frame_display` reuse cache covers same-buffer steady state; the only loss is one cold build on Ctrl+Tab to a never-prewarmed large buffer (covered by the existing per-paint cache miss reason logging).
- **Mouse hit-test + caret anchor reuse the paint cache.** `window_mouse_hit_test::client_to_buffer_position` and `window_caret_anchor::resolve_caret_display_line` previously cold-built a full `FrameDisplay` every `WM_MOUSEMOVE` / `WM_LBUTTONDOWN` / anchor call. On a 6000-line buffer each mouse move blocked the UI thread ~280 ms and each click ~930 ms (surfaced via `CONTINUITY_UI_TRACE`'s `stall100` lines). Both now reuse `last_painted_frame_display.clone()` (an `Arc::clone`) and consult cached `DisplayRowIndex` entries before building viewport specs. On stable-shape row-index miss, caret anchoring refreshes only the new caret line plus previous caret-reveal lines from the prior painted index; if that is unavailable it returns a source-line floor estimate. Mouse hit-test builds a source-line-floor fallback frame on row-index miss, so neither input handler emits `row_count_walker reason=caret_placement` or `reason=mouse_hit_test`. Caret lookup distinguishes unrealized viewport specs from folded lines through the row index and emits `caret_display_line_lookup` when trace is enabled.
- `Window::ensure_fresh_decorations` only sync-parses when the last keystroke was within 500 ms — the guard preserves its original purpose (masking worker-pool round-trips during typing bursts) while removing the sync re-parse from focus return / idle paint / mouse-driven repaint. The 256 KiB safety cap remains.
- **ε.5c plan classification shared by inline + worker (2026-05-16).** `crates/ui/src/window_projection_plan.rs` ships `classify_projection_build(ProjectionClassifyInputs<'_>) -> ProjectionBuildKind` — the single source of truth for how a paint should produce its `FrameDisplay`. Both the inline realization (`Window::realize_projection_build_kind`, same module) and the worker dispatch (`ProjectionBuildKind::to_worker_plan() -> Option<ProjectionPlan>`) consume the same kind, so a worker result can never disagree with what the inline path would have built. The classifier mirrors every branch of the prior inline ladder (wrap drift / line-count change / rope-history overrun → `Cold`; single newline → `Splice`; rope+decoration+selection-reveal drift → `Dirty`; covering cache with caret-only flip → `SelectionRebuild`; otherwise `CacheHit`). Cache-hit kinds map to `None` and skip worker dispatch outright — the worker can never beat a frame already realized at the current rope revision. With ε.3F splice + ε.3D decoration diff in place, worker submissions during typing are typically `Dirty` or `Splice` rather than `Cold`, so the worker stays close to the live rope state instead of redoing a full viewport build per keystroke. Trace detail now names the plan: `event:projection_worker_dispatch seq=… plan=cold | plan=dirty | plan=splice | plan=cache_hit`. Stale-frame fallback (accepting a worker result whose stamp lags by one stable-shape revision) is the ε.5d slice and intentionally deferred — the safety ladder still falls back to inline realization on any worker miss.
- **ε.5e early projection-worker dispatch on edit completion (2026-05-16).** `crates/ui/src/window_projection_early_dispatch.rs` ships `Window::try_dispatch_projection_worker_early(reason)`, called from `selection_dispatch::dispatch_selection_edit` and `selection::{insert_text, delete_back, delete_forward}_at_selections` immediately after `editor.apply_selection_edit` returns — before the next `WM_PAINT`. The helper rebuilds the same `PaintProjectionInputs` paint would build (decoration `transformed_through`, heading lines, folds, image reservations, projection metrics, viewport rows, font state) and runs them through the existing `current_projection_stamp` + `classify_projection_build` so the worker is asked for the exact frame paint will later accept. Pure helper `decide_early_dispatch_action(build_kind, stamp, last_stamp)` treats image reservations as stamp-validated projection input, so the active skip ladder is CacheHit → Dedupe → submit. `Window::last_early_dispatch_stamp` coalesces identical back-to-back submissions; the worker's bounded latest-wins recv coalesces an early-dispatch + post-paint pair into a single build. No state mutation: `cached_frame` passes as `None` (the prewarm cache requires `.take()` which would steal paint's state); the classifier picks Splice / Dirty / Cold against `last_painted_frame`. Preconditions mirror paint (`projection_worker`, `renderer`, `text_format`, snapshot) — any miss logs `event:projection_worker_early_dispatch submitted=false skip=skip_<reason>` and returns. The post-paint dispatch in `on_paint` is unchanged — it remains as the warm-up / fallback when an edit lands too close to paint for the worker to finish. Expected effect (verified by ε.5f below): typing-burst worker misses shift from `stamp_mismatch_rope_revision` toward `worker_hit` and `not_ready`.
- **ε.5f Win32Harness early-dispatch smoke (2026-05-16).** `crates/ui/tests/projection_worker_early_dispatch_smoke.rs` is the empirical proof of the ε.5e claim. It sets `CONTINUITY_UI_TRACE` to a tempdir-backed path before `Win32Harness::spawn` (so the `OnceLock` inside `paint_trace::ensure_initialized` captures it on every harness-spawned thread), injects 6000 lines as one `apply_edit`, runs two warmup paints, anchors trace assertions to the post-warmup byte offset, then `send_char('x')` through the production wndproc. The synchronous `event:projection_worker_early_dispatch submitted=true plan=dirty` event lands first; a bounded 2 s / 5 ms-interval poll loop then pumps `WM_PAINT` until the trace records `paint:frame_display:worker_hit` or fails at the deadline. Test is `#[ignore]`d to keep the pre-commit budget; run with `cargo test -p continuity-ui --test projection_worker_early_dispatch_smoke -- --ignored --nocapture`. The UI-thread frame-display-readiness perf gate lands in ε.5g (next bullet).
- **ε.5g UI-thread `WM_PAINT` → frame-ready perf gate (2026-05-16).** `crates/ui/src/paint_trace.rs` ships `PaintTrace::mark_since_start(stage, extra)` — emits `paint:<stage>` with cumulative duration from `PaintTrace::new()` rather than since the last `mark`; the inter-mark cursor stays untouched so this measurement does not perturb existing stage marks. `crates/ui/src/window_paint.rs::on_paint` calls `trace.mark_since_start("frame_ready", "source={worker_hit | inline_fallback} …")` on both branches of the `worker_outcome` match, right after `FrameDisplay` is in hand and *before* any renderer draw or `swap_chain.Present`, so the sample bounds exactly the UI-thread readiness window. Run with `cargo test -p continuity-ui --test perf_gate_wm_paint_to_frame_ready -- --ignored --nocapture`. With this gate green, ε.5 (off-UI-thread projection) is shipped end-to-end.
- **ε.3F multi-line-delete row-index splice (2026-05-17).** Symmetric to the multi-line-paste extension below. `DisplayRowIndex::dirty_after_rope_edits` now classifies any single-delta delete (`removed_bytes ≥ 1`, `inserted_bytes == 0`, `line_diff < 0`) as `RowDirty::Splice { at: byte_to_line(delta.at), removed: N + 1, inserted: 1, dirty: [at] }`, where `N = -line_diff` newlines were deleted. Subsumes the previous `removed_bytes == 1` single-newline-delete branch as `N = 1`. Covers backspace-over-multi-line-selection, delete-to-end-of-paragraph, multi-line cut. Belt-and-suspenders `at + N + 1 ≤ old_lines` guard falls back to `FullRebuild` on EOF / synthetic-empty surprises. The splice builder is generic over `splice.removed` (unchanged). - **Status-bar segment X snap to whole DIPs (2026-05-17).** `compute_layout` in `crates/render/src/status_bar.rs` rounds each segment's `left` / `right` to the nearest integer DIP. Without rounding, slot widths accumulate via `chars * font_size * 0.55` (generally fractional) so changing-digit text inside a slot shifted by sub-DIP amounts between paints — DirectWrite ClearType then sampled the same glyphs against a different sub-pixel grid each frame and the user saw it as "blurry" digits. With rounding, same-text repaints are pixel-identical; only the slot whose text actually changed re-rasterises, and it always lands on the same integer pixel column.
- **Status-bar transient suppression for high-frequency counters (2026-05-17).** `StatusMotionState::update` (`crates/ui/src/status_motion.rs`) checks `is_motion_eligible_kind(StatusBarSegmentKind)` before scheduling a 180 ms acknowledgement transient. `Position`, `Chars`, `Words`, `Lines`, `Selection`, `NumericSum` are *suppressed* — every keystroke advancing those would re-arm a `−3 DIP → 0` slide + `1.0 → 0.0` alpha fade *on top of* the steady-state draw at the steady Y, producing a perpetual ghost-offset double-image. `Encoding`, `LineEndings`, `Language`, `IdleStale`, `Chip`, `NoticeChip`, `PersistQueueChip` remain eligible — they change rarely enough that the motion reads as the intended tactile bump. See `motion.md` § Status Chips for the user-facing contract.
- **Stale-decoration byte-range transform.** `EditorHandle::rope_deltas_since(buffer_id, since_revision)` returns the chain of edits applied after a cached `Decorations` was computed. `Decorations::transformed_through(deltas, new_revision)` walks each span's byte range through the chain — spans entirely before any edit stay put, spans entirely after shift by the accumulated byte delta, spans that intersect any edit are dropped. The window paint path runs this whenever the cached decorations are stale, so painting against a multi-keystroke-stale snapshot keeps markdown styling aligned against the current rope instead of misplacing every span below the latest edit and snapping back when the worker catches up. The bounded per-buffer history (`DELTA_HISTORY_CAP = 512` entries) reports `covered=false` when the request asks for a revision older than the most-recently-evicted entry — in that case the paint drops the cached decorations (one undecorated frame, then the worker's next result reseeds the cache). Trace events: `decorations_state exact | transformed | dropped=history_overflowed | miss=cache_empty`.

## Steady-state memory targets
- ≤ 40 MB resident on a fresh empty session.
- Layout cache ≤ 10× visible lines per pane (a typical 4-pane 60-line viewport ≈ 2 400 cached layouts).
- One `Arc<DisplayMap>` per buffer per pane × revisions kept (snapshot lifetime is one paint frame; older `Arc`s drop immediately).

## Failure modes
- **Layout cache miss storm** (e.g. font reflow on theme reload) ⇒ paint stutter for one frame ⇒ acceptable; subsequent frames hit cache.
- **Decoration worker pool saturated** ⇒ stale styles for a paint frame ⇒ accepted-on-arrival rule keeps the editor coherent on the next paint.
- **Persist queue saturated** ⇒ coalesce policy fires ⇒ batch grows ≤ 8 MB before the core blocks; user keystrokes never lag.

## Anti-bloat patterns (spec §16)
- **Resist generic abstractions.** No `Backend` trait over rope flavors, no `Renderer` trait over graphics APIs, no `Platform` trait — Windows-only, ropey-only, D2D-only.
- **One concept per module.** A file growing past 600 lines is split by responsibility, not alphabet.
- **No serde_yaml.** Async-tainted; we don't ship an async runtime. TOML only.
